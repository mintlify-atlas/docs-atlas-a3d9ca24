## Introduction

ELI NoSQL is a custom NoSQL database language compiler with a Swing-based IDE. The system follows a classic compiler architecture with distinct phases: lexical analysis, syntax analysis, and interpretation/execution.

## Compilation Pipeline

The compilation and execution pipeline consists of four major phases:

```
Source Code (.eli)
      |
      v
  [Lexer] (JFlex)
      |
      v
   Tokens
      |
      v
  [Parser] (CUP)
      |
      v
 Semantic Actions
      |
      v
 [Interpreter]
      |
      v
 In-Memory Database + JSON Persistence
```

### Phase 1: Lexical Analysis
The **Lexer** (generated by JFlex from `Scanner.jflex`) reads source code and produces tokens:
- Keywords: `database`, `table`, `use`, `read`, `add`, `update`, `clear`, `export`
- Data types: `int`, `float`, `string`, `bool`, `array`, `object`
- Operators: `==`, `!=`, `>`, `<`, `>=`, `<=`, `&&`, `||`, `!`
- Literals: integers, floats, strings, booleans
- Identifiers and punctuation

### Phase 2: Syntax Analysis
The **Parser** (generated by CUP from `Parser.cup`) validates syntax and builds semantic actions:
- Validates grammar rules
- Calls interpreter methods directly during parsing
- Handles syntax error recovery
- Constructs filter expressions for queries

### Phase 3: Semantic Analysis & Execution
The **Interpreter** (`Modelo.Interprete.java`) executes commands immediately:
- Validates schema and types
- Executes DDL (Data Definition Language): `database`, `table`
- Executes DML (Data Manipulation Language): `add`, `update`, `clear`
- Executes DQL (Data Query Language): `read` with filters
- Manages in-memory database state

### Phase 4: Persistence
JSON-based persistence layer:
- Auto-saves database state after mutations
- Loads database from JSON on startup
- Exports query results to JSON files

## Package Structure

The project is organized into the following Java packages:

### `analisis` Package
Contains compiler front-end components:
- **`Lexer.java`** - Generated lexical analyzer (from JFlex)
- **`Parser.java`** - Generated parser (from CUP)
- **`sym.java`** - Token symbols (generated by CUP)
- **`Reportes.java`** - Error and token reporting

### `Modelo` Package
Contains the execution engine and data model:
- **`Interprete.java`** - Main interpreter and command executor
- **`BaseDatos.java`** - Database container class
- **`Tabla.java`** - Table with schema and records
- **`Registro.java`** - Individual record/row

### `compilador` Package
Contains the Swing IDE integration:
- GUI components
- Editor panel
- Console output
- Report visualization

## Data Flow Example

Consider this ELI NoSQL command:

```eli
read users {
  fields: name, age;
  filter: age >= 18 && status == "active";
};
```

**Step-by-step execution:**

1. **Lexer** tokenizes: `READ`, `IDENTIFIER("users")`, `LBRACE`, `FIELDS`, `COLON`, `IDENTIFIER("name")`, `COMMA`, `IDENTIFIER("age")`, ...

2. **Parser** matches `read_stmt` grammar rule:
   - Calls `Interprete.agregarCampoRead("name")`
   - Calls `Interprete.agregarCampoRead("age")`
   - Constructs filter string: `"(age>=18&&status==\"active\")"`
   - Calls `Interprete.establecerFiltro(...)`
   - Calls `Interprete.ejecutarRead("users")`

3. **Interpreter** executes:
   - Gets active database: `bdActiva`
   - Gets table: `bdActiva.getTabla("users")`
   - Iterates through `tabla.getRegistros()`
   - For each record, calls `evaluarFiltro(registro, filterExpr)`
   - Collects matching records
   - Formats output to console

4. **Filter Evaluator** parses the filter expression:
   - Splits on `&&` (AND operator)
   - Evaluates `age>=18` by comparing record field
   - Evaluates `status=="active"` by comparing record field
   - Returns `true` if both conditions match

## Key Design Decisions

### Immediate Execution
Unlike traditional compilers, ELI NoSQL uses an **interpreter pattern** that executes commands immediately during parsing. Each grammar production calls interpreter methods directly.

**Benefits:**
- Simpler architecture (no intermediate AST)
- Faster development cycle
- Immediate feedback to users

**Trade-offs:**
- Cannot perform multi-pass optimizations
- Limited static analysis capabilities
- Error recovery is more complex

### In-Memory Database
All data is stored in Java collections (`LinkedHashMap`, `ArrayList`) during execution:
- Fast access and manipulation
- Simple implementation
- Suitable for educational/prototyping purposes

**Persistence Strategy:**
- Auto-save after every mutation (`add`, `update`, `clear`)
- JSON format for human readability
- Load-on-demand when database is opened

### Schema Validation
Strict type checking at insertion/update time:
- Schema defined at table creation
- Type validation in `Interprete.validarTipo()`
- Supported types: `int`, `float`, `string`, `bool`, `array`, `object`

## Error Handling

The compiler reports three types of errors:

1. **Lexical Errors** - Reported by Scanner.jflex:179-182
   - Illegal characters
   - Malformed string literals

2. **Syntax Errors** - Reported by Parser.cup:12-38
   - Unexpected tokens
   - Missing semicolons or braces
   - Grammar violations

3. **Semantic Errors** - Reported by Interprete.java
   - Type mismatches
   - Undefined tables or databases
   - Missing required fields
   - Invalid filter expressions
